## I. THE ADDRESS SPACE

### The Full Lattice

12 Operators × 6 Axes × 7 Orders × 8 Colors = **4,032 base zip codes**

These are NOT organized by Order (the old weekday-locked model). They are organized by **training temperature** — a derived thermal value from the combination of all 4 dials. Think of 4,032 pie slices on a circle, sorted from cold to hot:

- **Cold:** 🖼 Restoration × 🏛 Basics × ⚪ Mindful = recovery, parasympathetic, mobility
- **Cool:** 🐂 Foundation × 🏛 Basics × ⚫ Teaching = technical, instructional, moderate
- **Warm:** 🦋 Hypertrophy × 🔨 Functional × 🟠 Circuit = moderate-high intensity, volume
- **Hot:** 🏟 Performance × 🪐 Challenge × 🔴 Intense = maximal effort, low volume, peak output

Adjacent slices on the circle have similar thermal character. The temperature is a derived value, not a stored field — it emerges from the 4-dial combination.

### All 5 Types Live at Every Zip Code

The 5 workout types (🛒 Push, 🪡 Pull, 🍗 Legs, ➕ Plus, ➖ Ultra) are NOT a dial dimension. They exist AT every zip code. Each of the 4,032 base addresses hosts all 5 types as available workout modes.

4,032 × 5 = **20,160 total workout addresses**

### The Revelator String for Workouts

```
[ZIP] ± [TYPE] | [BLOCKS...]
```

- **ZIP** = 4-digit base address (Operator · Axis · Order · Color) — the training temperature
- **± TYPE** = one of 5 workout Diameters (🛒🪡🍗➕➖) — tail selection picks the workout
- **| BLOCKS** = the intra-workout program flow using Block Diameters

Example: `🤌🏛⛽🔵 ± 🪡 | ♨️▶️🧈🧈🧩🌋🪫` = Builder-Basics-Strength-Structured Pull workout with warm-up, primer, two main sets, accessory, finisher, cool-down.

## II. THE TWO-PAN SCALE (Applied to Workouts)

### Left Pan — System Map

4,032 base addresses. Each address has its own base-7 octave counter tracking the user's personal depth at that coordinate. This is the universal map. Everyone shares the same lattice. Your altitude at each point is your own.

All 7 Orders are available at every address on every day. Orders are not locked to calendar days at this layer. The base-7 octave tracks cumulative passage through them.

### Right Pan — Diameter Ledgers

61 Diameters × 4,032 addresses = 245,952 combinations. Each position has 7 Order rows (Trivium + Quadrivium), each with its own base-7 octave counter.

245,952 × 7 = **1,720,664 total tracking cells**

### The Fulcrum — 427 Active Addresses

427 / 61 = 7. Exactly. The Zipf effective support at criticality (427), divided by the Diameter count (61), equals the Order count (7). Three independently derived numbers.

The 427 daily district is the system's surfaced recommendation — the addresses it highlights today based on the coprime rotation, the user's entropy balance, and the scale's current tilt. The district shifts as the user moves.

### Per-Exercise Tracking

Each exercise in the 2,085 library also gets a right-pan table:

61 Diameters × 7 Orders = **427 cells per exercise**

The number regenerates at the exercise level. Total across the library: 2,085 × 427 = **890,295 exercise tracking cells**.

## III. THE THREE-LAYER WORKOUT ADDRESS

### Layer 1 — The ZIP (Training Temperature)

One of 4,032 base addresses. Sets the thermal context:
- Loading parameters (from Order)
- Movement lens (from Axis)
- Phase character (from Order)
- Session register (from Color)
- Agent voice (from Operator)

The user can free-spin to any zip at any time. The 427 daily district is a recommendation, not a boundary.

### Layer 2 — The TAIL (Workout Type)

One of 5 type Diameters: 🛒 Push · 🪡 Pull · 🍗 Legs · ➕ Plus · ➖ Ultra

The tail picks WHICH workout you're doing at this zip. This is the first ± selection in the Revelator string. All 5 types are available at every zip code.

### Layer 3 — The BLOCKS (Intra-Workout Programming)

The 22 Block Diameters are the ONLY Diameters used inside a workout. They are structural templates, not exercises. Each block defines:
- What information stream it carries
- What exercise variables it surfaces
- What input fields it offers
- How the user engages with that segment

## IV. BLOCK BEHAVIOR — THE INTRA-WORKOUT LANGUAGE

### Blocks Are Engagement Modes, Not Exercises

The user taps a block icon → it **spins** to another block. Same exercise position, different engagement template. The block doesn't change WHAT. It changes HOW.

Examples of block spinning at the main lift position:
- 🧈 Canon = straight working sets, bread & butter
- 🪜 Progression = same exercise, ladder/ramp scheme
- 🪞 Facade = same exercise, form focus, lower load, higher intentionality
- 🏖 Sandbox = experimental mode, loosened constraints
- 🎱 Rotation = ARAM house dare (see Section V)

### Block Information Streams

Each block carries its own exercise variables and input fields:
- 🧈 Canon: sets × reps × load × rest. Standard strength training inputs.
- 🎼 Composition: exercise A + exercise B, superset/circuit format, transition time.
- 🪜 Progression: starting load, increment per set, target RPE ceiling.
- 🌋 Gutter: time cap, AMRAP or density target, minimal rest.
- 🧬 Imprint: tempo prescription, eccentric/concentric timing, cue focus.
- 🏖 Sandbox: open fields, no constraints enforced, full user control.
- ♨️ Vestibule: movement prep sequence, tissue warmth check, mobility flow.

The block defines the FRAME. The exercise fills the frame. The user can spin the frame without changing the exercise, or change the exercise without spinning the frame.

## V. 🎱 ARAM — THE HOUSE DARE

### What Happens When 🎱 Appears

🎱 can appear at any point in a workout — mid-session, as a finisher, at the start. When it appears:

1. **Everything remaining REDACTS.** Like a scratch-off lottery ticket. The user can't see what's coming. All subsequent blocks are hidden.

2. **The user sees two options:** SPIN AWAY or GO.

3. **SPIN AWAY** → 🎱 dissolves. Normal workout resumes. Blocks un-redact. No penalty.

4. **GO** → Live event begins:

### The Live Event

- The zip dial spins — new address, house-picked
- The system **DEALS** exercises one at a time
- The user can't see what's next
- **Z-pad is the input surface** (5 workout type emojis):
  - 🛒 = "hit me with push"
  - 🪡 = "hit me with pull"
  - 🍗 = "hit me with legs"
  - ➕ = "hit me with plus"
  - ➖ = "hit me with ultra"
- Tapping a type = **biasing the house** ("more of this")
- The system deals based on the user's history, exercise families, engagement gaps, and current entropy
- It could be: a test (checking capacity at unfamiliar addresses), novelty exploration (exercises never tried), intensity probe (pushing an edge), or play (pure variety)

### System Control During ARAM

- **Rest timers are SYSTEM-CONTROLLED.** The house sets the pace. The user doesn't choose rest periods. This is the one moment the user gives up temporal autonomy — like a trainer controlling the clock.
- **Time between "done" clicks is tracked.** This is implicit RPE data.
- **Start-to-completion time is tracked.** Session density measured automatically.

### Tap Out

The user can **tap out at any point.** No penalty. The event ends. Whatever was completed compiles normally.

### Resolution

When the 🎱 event ends (tap out or completion), the 🎱 block **retroactively resolves** into the block that best describes what happened. If the user did heavy singles, it resolves to 🎯 Thesis. If it was a burnout circuit, it resolves to 🌋 Gutter. If it was exploratory variety, it resolves to 🌎 Survey.

The training record stays structurally clean. The system knows what happened even though the user experienced it as chaos.

### Why 🎱 Is Load-Bearing Architecture

1. **SOC Algorithm's primary user-facing instrument.** Prevents entropy collapse (over-specialization) through play, not prescription.
2. **Hazard-icon exercises enter naturally.** The house can deal exercises the user has never done. No opt-in friction — it's in the flow. The family map enriches automatically.
3. **Retroactive block resolution** keeps the training record clean.
4. **Z-pad bias = minimal UI, maximum signal.** Five buttons. Each tap says "more of this." Three competing signals (user request, gap analysis, novelty) produce the deal.
5. **Rest timer control = the trainer setting the pace.** The play-day energy. You're not programming, you're responding.
6. **Data goldmine for the right pan.** Every 🎱 session touches exercises and addresses the user wouldn't normally pick. The 427-cell tracking table gets enriched exactly where it was sparse.
7. **This is how Jake trains clients.** "Surprise me" day. Show up, read their energy, deal them something unexpected, control the rest. They leave feeling like they played a game but got the best stimulus of the week. Formalized.

## VI. EXERCISE SELECTION — OPEN FIELD + HAZARD

### When the User Wants to Change an Exercise

The user taps the exercise (not the block). The system shows available movements in two tiers:

**Open field (no icon):** Exercises the system's math knows fit well from a programming standpoint. These are movements within the user's established families, at appropriate difficulty, with sufficient history for the system to program intelligently.

**Hazard icon (⚠️):** Exercises the system doesn't have enough data on. Not wrong. Not bad. Not dangerous. **Unknown to the current math.** The system can't program them optimally yet because it hasn't seen the user do them.

Opting into a hazard exercise:
- Enriches the user's exercise library
- Expands the family map (new connections, new transfer ratios)
- Opens the door to better programming mathematically
- The unknown becomes known through use

**No exercise is blocked.** The hazard icon is information, not permission. The user makes the choice. The system metabolizes the result either way.

### Exercise Family Architecture

2,085 exercises organized in **base-7 family trees:**

- **Root exercise** = family founder (e.g., Barbell Back Squat for the Squat family)
- **Branches** at base-7 tiers: variants (.1-.7), progressions (.1-.7 at next depth), regressions, equipment swaps
- **Transfer ratios** link family members (0.55-1.05 range): how much strength/skill transfers between related exercises
- **15 major families** documented, covering the full movement vocabulary

Each exercise carries:
- 61 × 7 = 427 tracking cells (right-pan octave table)
- Family membership and transfer ratios to siblings
- Axis affinity and Order affinity scores (-8 to +8)
- Equipment tier (0-5)
- Movement pattern and subpattern
- Primary movers, secondary movers, stabilizers, joint actions

## VII. THE NO-DEVIATIONS PRINCIPLE

### The Bike Lock

The 427 daily district is a surfaced recommendation, not a boundary. The user free-spins to any of the 4,032 addresses at any time. There is no punishment for going elsewhere. There is no deviation. The compiler processes every visit identically:

- Planned or unplanned
- Within the 427 or outside it
- Open field exercise or hazard exercise
- Normal block or 🎱 ARAM
- Full session or tap-out

Every visit produces: an octave tick, a Diameter engagement record, an entropy update, and a family map enrichment.

### 73% Entropy Target — Moving Average, Not Quota

The critical entropy H* = 8.74 bits = 73% of H_max is a moving average across cycles, not a daily target. One day of exploration doesn't break criticality. One week of specialization doesn't either. The system breathes toward the target across mesocycles.

### Planned Layer Molts to Actual Layer

Trainer-architected planned sessions are locked columns on the calendar that don't move. But the user can add, substitute, skip, or explore freely around them. Skipped planned sessions remain as unvisited addresses — no penalty, no compliance score. The planned layer molts to fit the actual layer over time as the rotation engine incorporates the user's real navigation patterns.

## VIII. THE ROTATION ENGINE — REVISED

### What Carries Over

The coprime structure is **validated** by the research:
- 5-day Type cycle × 7-day Order mapping = 35-day supercycle (confirmed by Round 3 lattice analysis)
- 12-month Axis rotation = implicit annual periodization
- Color = user's one dial (8 options)
- Grand period of the lattice = LCM(12, 6, 7, 8) = 168

### What Changes

- **Orders are NOT locked to weekdays** at the lattice layer. All 7 Orders are available at every address on every day. The weekday-Order mapping is a DEFAULT RECOMMENDATION, not a constraint. The rotation engine surfaces the recommendation; the user can override.
- **The 427 daily district** replaces the old single-zip-per-day model. The system surfaces a neighborhood of recommended addresses based on the coprime rotation, entropy balance, and scale tilt.
- **Training temperature** organizes the 4,032 circle. The rotation engine navigates temperature, not just Order.
- **All 5 Types at every zip.** The Type cycle determines which type is RECOMMENDED, not which is available.

### Session Thresholds (from Round 4)

- **~2 sessions/week:** Maintenance. Entropy holds but doesn't grow.
- **~5 sessions/week:** Criticality. Entropy at the 73% target. 427 effective addresses.
- **~10+ sessions/week:** Risk of over-dispersion. Entropy too high. System recommends depth over breadth.
